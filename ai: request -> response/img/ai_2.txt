Полная схема: от запроса пользователя до ответа модели

╔══════════════════════════════════════════════════════════════════╗
║  1. ВВОД ПОЛЬЗОВАТЕЛЯ                                          ║
║     "Расскажи анекдот про кота"                                 ║
╚════════════════════════════╤═════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════╗
║  2. ТОКЕНИЗАЦИЯ                                                 ║
║                                                                  ║
║  Текст разбивается на токены по словарю модели (BPE алгоритм)   ║
║                                                                  ║
║  "Расскажи анекдот про кота"                                    ║
║       ↓                                                          ║
║  ["Рас", "скажи", " анек", "дот", " про", " кота"]             ║
║                                                                  ║
║  Каждый токен получает числовой ID из словаря:                   ║
║  [4821, 9903, 7712, 3304, 1082, 8845]                           ║
║                                                                  ║
║  Как работает BPE (Byte-Pair Encoding):                          ║
║  ─────────────────────────────────────                           ║
║  Словарь модели — это таблица поиска, где каждому фрагменту      ║
║  текста сопоставлено число. Вместо хранения целых слов,          ║
║  BPE разбивает слова на частые подслова (фрагменты).             ║
║  Это позволяет обрабатывать даже незнакомые слова —              ║
║  они просто разобьются на известные части.                       ║
║  Для слов, которых вообще нет в словаре, резервируется           ║
║  специальный токен [UNK] (unknown).                              ║
║                                                                  ║
║  Специальные токены:                                             ║
║  ─────────────────────────────────────                           ║
║  Помимо обычных токенов, добавляются служебные:                  ║
║  [BOS] — начало последовательности (Begin of Sequence)           ║
║  [EOS] — конец последовательности (End of Sequence)              ║
║  [PAD] — заполнитель для выравнивания длины                      ║
║  Они дают модели дополнительный контекст о структуре ввода.      ║
╚════════════════════════════╤═════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════╗
║  3. ЭМБЕДДИНГ (Embedding)                                       ║
║                                                                  ║
║  Каждый token ID превращается в вектор — массив чисел,          ║
║  который кодирует СМЫСЛ токена в многомерном пространстве       ║
║                                                                  ║
║  ID 4821 → [0.12, -0.45, 0.78, 0.03, ..., -0.21]  (2048+ чисел)║
║  ID 9903 → [0.56, 0.11, -0.33, 0.89, ..., 0.44]                ║
║  ...                                                             ║
║                                                                  ║
║  Зачем нужны эмбеддинги:                                        ║
║  ─────────────────────────────────────                           ║
║  Числовой ID (например 4821) — это просто индекс, он ничего     ║
║  не говорит о смысле слова. Эмбеддинг же передаёт смысл         ║
║  в сжатом формате: похожие по значению слова получают            ║
║  похожие векторы (близкие точки в пространстве).                 ║
║                                                                  ║
║  Изначально эмбеддинги — случайные числа. Осмысленные            ║
║  представления формируются в процессе обучения модели.            ║
║                                                                  ║
║  Проблема: на этом этапе эмбеддинг одного и того же слова        ║
║  ОДИНАКОВ в любом контексте. Например "light" (свет) и           ║
║  "light" (лёгкий) пока имеют один и тот же вектор.              ║
║  Решение этой проблемы — задача трансформера (шаг 4).            ║
║                                                                  ║
║  + Positional Encoding — добавляется информация о ПОЗИЦИИ       ║
║    каждого токена в предложении (модель должна знать порядок)    ║
║                                                                  ║
║    Это отдельный набор эмбеддингов, представляющих положение     ║
║    каждого токена в последовательности. Он комбинируется         ║
║    (складывается) с эмбеддингом токена.                           ║
║    Без этого модель не отличила бы "кот ел рыбу"                 ║
║    от "рыба ела кота" — набор токенов одинаков.                  ║
╚════════════════════════════╤═════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════╗
║  4. ТРАНСФОРМЕР (Transformer)                                    ║
║                                                                  ║
║  Ядро модели. Набор слоёв (у Gemini — десятки слоёв),           ║
║  каждый слой содержит:                                           ║
║                                                                  ║
║  Главное преимущество трансформеров — способность обрабатывать   ║
║  длинные зависимости в тексте (связь между словами, которые      ║
║  далеко друг от друга) и делать это параллельно, а не            ║
║  последовательно, как старые архитектуры (RNN/LSTM).             ║
║                                                                  ║
║  ┌─────────────────────────────────────────────────────────┐    ║
║  │  Self-Attention (механизм внимания)                      │    ║
║  │                                                          │    ║
║  │  Каждый токен "смотрит" на все остальные токены           │    ║
║  │  и определяет, какие из них важны для текущего контекста  │    ║
║  │                                                          │    ║
║  │  "кота" обращает внимание на "анекдот" и "расскажи"      │    ║
║  │  → понимает: нужен юмор про кота                         │    ║
║  │                                                          │    ║
║  │  Как это работает технически:                             │    ║
║  │  ────────────────────────                                │    ║
║  │  Матрица эмбеддингов пропускается через три независимых   │    ║
║  │  линейных слоя (матричных умножения), получая:            │    ║
║  │                                                          │    ║
║  │  Q (Query)  — "что я ищу?" — запрос токена               │    ║
║  │  K (Key)    — "что я содержу?" — ключ токена             │    ║
║  │  V (Value)  — "что я отдам?" — значение токена           │    ║
║  │                                                          │    ║
║  │  1) Вычисляем скалярное произведение Q × K^T              │    ║
║  │     → матрица "оценок внимания" (attention scores)       │    ║
║  │     Чем больше произведение — тем больше сходство         │    ║
║  │     между двумя токенами                                 │    ║
║  │                                                          │    ║
║  │  2) Масштабирование: делим на √d (размерность ключа)     │    ║
║  │     Без этого большие числа после умножения дадут         │    ║
║  │     слишком "острое" распределение, что приведёт          │    ║
║  │     к проблеме исчезающего градиента при обучении         │    ║
║  │                                                          │    ║
║  │  3) Softmax — превращаем оценки в веса (0..1),           │    ║
║  │     которые в сумме дают 1                                │    ║
║  │                                                          │    ║
║  │  4) Умножаем веса на V → получаем новые                   │    ║
║  │     контекстуализированные эмбеддинги                     │    ║
║  │     (теперь каждый токен "впитал" информацию              │    ║
║  │      от релевантных соседей)                              │    ║
║  │                                                          │    ║
║  │  Итого формула:                                          │    ║
║  │  Attention(Q,K,V) = softmax(Q × K^T / √d) × V           │    ║
║  │                                                          │    ║
║  │  Self-attention означает, что Q, K, V получены            │    ║
║  │  из ОДНОЙ и той же входной матрицы — модель ищет          │    ║
║  │  закономерности внутри своих же входных данных.           │    ║
║  │                                                          │    ║
║  │  В декодерах (GPT, Claude) используется маскированный     │    ║
║  │  self-attention: каждый токен видит только себя и         │    ║
║  │  предыдущие токены. Будущие позиции маскируются через     │    ║
║  │  нижнетреугольную матрицу (замена на -∞ → после           │    ║
║  │  softmax даёт 0). Модель не может "подглядывать вперёд". │    ║
║  └─────────────────────────────────────────────────────────┘    ║
║                         │                                        ║
║                         ▼                                        ║
║  ┌─────────────────────────────────────────────────────────┐    ║
║  │  Multi-Head Attention (многоголовое внимание)            │    ║
║  │                                                          │    ║
║  │  На практике параллельно запускается несколько блоков     │    ║
║  │  self-attention (голов), чтобы модель одновременно        │    ║
║  │  улавливала РАЗНЫЕ типы связей между токенами:           │    ║
║  │  одна голова следит за грамматикой, другая — за смыслом,  │    ║
║  │  третья — за тоном и т.д.                                │    ║
║  │                                                          │    ║
║  │  Размерность каждой головы = размер эмбеддинга / кол-во  │    ║
║  │  голов (чтобы суммарный размер не менялся).              │    ║
║  │                                                          │    ║
║  │  Результаты всех голов конкатенируются и пропускаются     │    ║
║  │  через линейный слой, который комбинирует информацию      │    ║
║  │  от каждой головы в единое представление.                 │    ║
║  └─────────────────────────────────────────────────────────┘    ║
║                         │                                        ║
║                         ▼                                        ║
║  ┌─────────────────────────────────────────────────────────┐    ║
║  │  Feed-Forward Network (нейросеть прямого прохода)        │    ║
║  │                                                          │    ║
║  │  Обрабатывает результат attention, извлекает              │    ║
║  │  более сложные закономерности и "знания"                 │    ║
║  │                                                          │    ║
║  │  Это двухслойная нейросеть, которая применяется           │    ║
║  │  НЕЗАВИСИМО к эмбеддингу каждого токена.                  │    ║
║  │  Её задача — добавить дополнительные изучаемые            │    ║
║  │  параметры, чтобы контекстуальные эмбеддинги             │    ║
║  │  были хорошо разделены и распределены в пространстве.     │    ║
║  │  Функция активации — GeLU (или другая, зависит от        │    ║
║  │  архитектуры).                                           │    ║
║  └─────────────────────────────────────────────────────────┘    ║
║                         │                                        ║
║                         ▼                                        ║
║  ┌─────────────────────────────────────────────────────────┐    ║
║  │  Layer Normalization + Skip Connections                   │    ║
║  │                                                          │    ║
║  │  Layer Norm — нормализует значения в каждом слое,        │    ║
║  │  не позволяя им стать слишком большими или маленькими.   │    ║
║  │  Без этого глубокие модели страдают от исчезающего       │    ║
║  │  или взрывающегося градиента (обучение ломается).        │    ║
║  │                                                          │    ║
║  │  Skip Connections (остаточные связи) — вход слоя          │    ║
║  │  прибавляется к его выходу (как в ResNet).               │    ║
║  │  Это позволяет градиенту "протекать" через слои          │    ║
║  │  напрямую, не затухая — критично для обучения            │    ║
║  │  моделей с десятками/сотнями слоёв.                      │    ║
║  │                                                          │    ║
║  │  Современный подход (pre-layer norm):                     │    ║
║  │  нормализация ставится ДО attention и FFN                │    ║
║  │  (а не после, как в оригинальной статье).                │    ║
║  └─────────────────────────────────────────────────────────┘    ║
║                                                                  ║
║  Это повторяется N раз (слой за слоем), каждый раз углубляя     ║
║  понимание контекста                                             ║
║                                                                  ║
║  После прохождения всех слоёв эмбеддинги становятся              ║
║  контекстуализированными — теперь "light" в контексте            ║
║  "light bag" и "light in the room" имеют РАЗНЫЕ векторы.        ║
╚════════════════════════════╤═════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════╗
║  5. ЛОГИТЫ (Logits)                                              ║
║                                                                  ║
║  На выходе трансформера — один вектор, который проецируется      ║
║  на ВЕСЬ словарь. Получаем сырую оценку для каждого токена:     ║
║                                                                  ║
║  Берётся контекстуализированный эмбеддинг ПОСЛЕДНЕГО токена     ║
║  (именно он аккумулировал весь контекст через attention)         ║
║  и умножается на матрицу весов размером [d × vocab_size].       ║
║  Результат — вектор длиной = размер словаря:                     ║
║                                                                  ║
║  "Идёт"    →  5.8                                                ║
║  "Жил"     →  5.2                                                ║
║  "Кот"     →  4.1                                                ║
║  "Однажды" →  3.9                                                ║
║  "The"     → -2.1                                                ║
║  "xyz"     → -9.4                                                ║
║  ... (30 000+ оценок — по одной на каждый токен в словаре)      ║
║                                                                  ║
║  Это ещё НЕ вероятности, просто числа                            ║
╚════════════════════════════╤═════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════╗
║  6. СЭМПЛИРОВАНИЕ (выбор следующего токена)                      ║
║                                                                  ║
║  После получения логитов нужно выбрать один токен из             ║
║  30 000+ кандидатов. Объём данных огромен, поэтому               ║
║  применяется цепочка фильтров для сужения выбора:                ║
║                                                                  ║
║  6.1  Temperature                                                ║
║  ─────────────────                                               ║
║  Каждый логит делится на temperature:                             ║
║  logit / temperature                                             ║
║                                                                  ║
║  Низкая (0.1) → разница усиливается → предсказуемый ответ       ║
║  Высокая (2.0) → разница сглаживается → креативный ответ        ║
║                                                                  ║
║  Temperature влияет на "остроту" распределения вероятностей:      ║
║  при T→0 модель почти всегда выбирает самый вероятный токен,    ║
║  при T→∞ выбор становится почти случайным.                       ║
║                         │                                        ║
║                         ▼                                        ║
║  6.2  Top-K                                                      ║
║  ─────────────────                                               ║
║  Оставляем только K самых вероятных токенов                      ║
║  (topK=64 → берём 64, остальные 29 936 отбрасываем)             ║
║                                                                  ║
║  Фильтрация по количеству: оставляем фиксированное число        ║
║  лучших кандидатов на основе их логитов.                         ║
║                         │                                        ║
║                         ▼                                        ║
║  6.3  Top-P (Nucleus Sampling)                                   ║
║  ─────────────────                                               ║
║  Из оставшихся берём минимальный набор, чья суммарная            ║
║  вероятность ≥ 95% (topP=0.95)                                   ║
║                                                                  ║
║  Фильтрация по порогу вероятности: количество кандидатов         ║
║  плавающее — в одном случае останется 5 токенов,                 ║
║  в другом 50, в зависимости от того, насколько модель            ║
║  "уверена" в следующем слове.                                    ║
║                         │                                        ║
║                         ▼                                        ║
║  6.4  Softmax → Вероятности                                      ║
║  ─────────────────                                               ║
║  Преобразуем отфильтрованные логиты в вероятности:               ║
║                                                                  ║
║  "Идёт"    → 42%                                                 ║
║  "Жил"     → 31%                                                 ║
║  "Кот"     → 18%                                                 ║
║  "Однажды" →  9%                                                 ║
║                         │                                        ║
║                         ▼                                        ║
║  6.5  Случайный выбор (sampling)                                 ║
║  ─────────────────                                               ║
║  Бросаем "кубик" с учётом вероятностей                           ║
║  (выборка из полиномиального распределения)                      ║
║  Результат: "Идёт"                                               ║
╚════════════════════════════╤═════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════╗

║                                                                  ║

╔══════════════════════════════════════════════════════════════════╗
║  8. АВТОРЕГРЕССИОННЫЙ ЦИКЛ                                       ║
║                                                                  ║
║  Выбранный токен добавляется к входной последовательности:       ║
║                                                                  ║
║  Итерация 1: "Расскажи анекдот про кота"          → "Идёт"     ║
║  Итерация 2: "Расскажи анекдот про кота Идёт"     → "кот"      ║
║  Итерация 3: "Расскажи анекдот про кота Идёт кот" → "по"       ║
║  ...                                                             ║
║                                                                  ║
║  Каждая итерация проходит ЗАНОВО шаги 3 → 4 → 5 → 6 → 7       ║
║  (с учётом KV-cache для ускорения)                               ║
║                                                                  ║
║  KV-Cache:                                                       ║
║  ─────────────────                                               ║
║  При генерации каждого нового токена модель пересчитывает        ║
║  attention для всей последовательности. Но K и V для уже         ║
║  обработанных токенов НЕ МЕНЯЮТСЯ — их можно кэшировать.        ║
║  Новый токен вычисляет свой Q и сравнивает его с сохранёнными   ║
║  K и V, вместо того чтобы пересчитывать всё с нуля.            ║
║  Это значительно ускоряет генерацию.                             ║
╚════════════════════════════╤═════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════╗
║  9. ДЕТОКЕНИЗАЦИЯ                                                ║
║                                                                  ║
║  Набранные токены склеиваются обратно в текст:                   ║
║                                                                  ║
║  ["Идёт", " кот", " по", " крыше", "..."]                      ║
║       ↓                                                          ║
║  "Идёт кот по крыше..."                                          ║
║                                                                  ║
║  Этот текст отдаётся пользователю                                ║
║  (при streaming — токен за токеном по мере генерации)            ║

═══════════════════════════════════════════════════════════════════

  Трансформеры бывают трёх типов в зависимости от того,
  какие блоки используются:

  ┌─────────────────────────────────────────────────────────────┐
  │  A. Энкодер (Encoder)                                       │
  │                                                             │
  │  Видит ВСЮ входную последовательность целиком.              │
  │  Задача — создать контекстуализированные эмбеддинги         │
  │  для последующих задач: классификация текста,               │
  │  распознавание сущностей и т.д.                             │
  │                                                             │
  │  Пример: BERT                                               │
  │  Применение: "Этот отзыв положительный или отрицательный?"  │
  └─────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────┐
  │  B. Декодер (Decoder)                                       │
  │                                                             │
  │  Видит только текущий и ПРЕДЫДУЩИЕ токены                   │
  │  (будущие замаскированы). Генерирует текст токен за токеном. │
  │  Именно эта архитектура используется в ChatGPT, Claude.     │
  │                                                             │
  │  Пример: GPT, Claude, LLaMA                                 │
  │  Применение: генерация текста, диалоги, написание кода      │
  └─────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────┐
  │  C. Энкодер-Декодер                                         │
  │                                                             │
  │  Энкодер читает вход целиком → создаёт промежуточное        │
  │  представление → декодер генерирует выход на его основе.     │
  │  Декодер при этом использует cross-attention:               │
  │  K и V берутся из энкодера, Q — из декодера.               │
  │  (сравни с self-attention, где Q, K, V — из одного входа)  │
  │                                                             │
  │  Пример: T5, оригинальный Transformer                       │
  │  Применение: перевод, суммаризация, ответы на вопросы       │
  └─────────────────────────────────────────────────────────────┘
