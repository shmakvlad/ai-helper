# Что это делает
Преобразует сырые инженерные артефакты — комментарии в коде, архитектурные диаграммы, проектные документы и переписку в Slack — в структурированную техническую документацию, адаптированную под разные аудитории (разработчики, операционная команда, руководство).

---

## Быстрый старт

**Шаг 1: Скачай шаблон**
Нажми Download выше, чтобы получить файл CLAUDE.md.

**Шаг 2: Собери инженерные материалы**
Подготовь файлы с кодом, проектные документы, архитектурные диаграммы и любые релевантные обсуждения команды.

**Шаг 3: Начни работу**
```
claude
```
Скажи: «Создай техническую документацию для нашего сервиса обработки платежей на основе этих проектных документов и файлов с кодом. Целевая аудитория: новые инженеры, которые онбордятся в команду.»

---

## Типы документации

| Тип | Аудитория |
|-----|-----------|
| Обзор архитектуры | Инженеры и архитекторы — дизайн системы и принятые решения |
| Runbook | Операционная команда — деплой, мониторинг и устранение неполадок |
| Гайд по онбордингу | Новые инженеры — как работает система и как вносить вклад |
| Справочник API | Разработчики интеграций — эндпоинты, схемы, примеры |
| Записи решений | Будущая команда — почему были приняты те или иные решения (ADR) |
| Диаграмма системы | Все аудитории — визуальное представление с контекстом |

---

## Советы

- **Начинай с «зачем»:** прежде чем объяснять, как работает система, объясни, зачем она существует
- **Используй реальные примеры:** фрагменты кода и API-вызовы из реальной системы
- **Документируй решения, а не только дизайн:** ADR предотвращают повторные споры об одном и том же
- **Держи диаграммы простыми:** одна диаграмма — одна концепция. Хорошо работает модель C4

- «Сгенерируй runbook для деплоя и эксплуатации [сервис]»
- «Составь гайд по онбордингу для новых инженеров в [команда]»

## Устранение проблем

**Документация слишком детальная для аудитории**
Скажи: «Напиши для человека, который разбирается в технологиях, но не знаком с этой конкретной системой.»

**Информация разбросана по разным источникам**
Спроси: «Создай единый источник истины, который консолидирует эти материалы.»

**Документация сразу устаревает**
Уточни: «Генерируй документацию из кода там, где это возможно. Документируй стабильные концепции, а не изменчивые детали.»
