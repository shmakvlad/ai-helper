┌──────────────────────────────────────────────────────────────────────────────┐
│  1.1  ЧТО ТАКОЕ ПАРАМЕТРЫ?                                                 │
└──────────────────────────────────────────────────────────────────────────────┘

  Параметры — это веса (weights) и смещения (biases) внутри матриц, из
  которых состоит нейросеть. Каждый параметр — число с плавающей запятой,
  например 0.0037 или −1.228.

  У GPT-4 их порядка триллиона, у Claude — сотни миллиардов.

  Они НЕ содержат «знаний» в виде записей вроде базы данных.
  Вместо этого они кодируют статистические закономерности языка —
  как распределённая голограмма, где каждый фрагмент несёт часть
  всей картины.

  ╔═══════════════════════════════════════════════════════════════════════╗
  ║  АНАЛОГИЯ «МИКШЕРНЫЙ ПУЛЬТ»                                        ║
  ║                                                                     ║
  ║  Представь гигантский микшерный пульт в студии звукозаписи.         ║
  ║  На нём миллиарды ручек-крутилок. Каждая ручка — один параметр.     ║
  ║  Сами по себе эти числа бессмысленны. Но когда они настроены        ║
  ║  в совокупности, пульт превращает любой входной сигнал (запрос)      ║
  ║  в осмысленный выходной (ответ).                                    ║
  ╚═══════════════════════════════════════════════════════════════════════╝


┌──────────────────────────────────────────────────────────────────────────────┐
│  1.2  КАК ПРОИСХОДИТ ОБУЧЕНИЕ?                                              │
└──────────────────────────────────────────────────────────────────────────────┘

  Обучение идёт в три стадии:


  ── Стадия 1: Pre-training (предобучение) ──────────────────────────────

  Модели скармливают терабайты текста из интернета, книг, статей, кода.
  Задача проста: по предыдущим словам предскажи следующее.

  Модель выдаёт прогноз, сравнивает с реальным следующим словом,
  вычисляет ошибку (loss), и алгоритм обратного распространения
  (backpropagation) чуть-чуть подкручивает каждую из миллиардов
  «ручек» так, чтобы в следующий раз ошибка была меньше.

  Это повторяется триллионы раз.

  Схема одной итерации:
  ┌────────────────────────────────────────────────────────────────────┐
  │                                                                    │
  │  Данные ──→ [Предскажи следующее слово] ──→ Ошибка (loss)        │
  │    ↑                                            │                  │
  │    │                                            ▼                  │
  │    │                                     Backpropagation           │
  │    │                                            │                  │
  │    │                                            ▼                  │
  │    └──────────────── Обновить параметры ◄───────┘                  │
  │                                                                    │
  │                    × триллионы раз                                 │
  └────────────────────────────────────────────────────────────────────┘

  ╔═══════════════════════════════════════════════════════════════════════╗
  ║  АНАЛОГИЯ «СКУЛЬПТОР»                                               ║
  ║                                                                     ║
  ║  Представь скульптора, который начинает с бесформенной глыбы        ║
  ║  мрамора. Каждый проход по тексту — один удар резцом. Ни один      ║
  ║  удар не создаёт скульптуру. Но после миллиардов ударов            ║
  ║  проступает форма. Параметры — это не «запомненные факты»,         ║
  ║  а рельеф скульптуры, который определяет, как свет (запрос)        ║
  ║  отразится от поверхности (ответ).                                  ║
  ╚═══════════════════════════════════════════════════════════════════════╝


  ── Стадия 2: Fine-tuning + RLHF ──────────────────────────────────────

  После предобучения модель умеет продолжать текст, но не умеет
  «общаться». Она может выдать Википедию, но не может вежливо ответить
  на вопрос.

  Дообучение происходит в два этапа:

  1) Supervised Fine-Tuning (SFT):
     Модель дообучают на парах «вопрос → хороший ответ»,
     написанных людьми-аннотаторами.

  2) RLHF (Reinforcement Learning from Human Feedback):
     Людям показывают несколько вариантов ответа модели, они
     ранжируют их от лучшего к худшему. На этих рангах обучается
     отдельная «модель вознаграждения» (Reward Model), которая
     затем направляет основную модель предпочитать полезные,
     безопасные и честные ответы.

  Схема:
  ┌────────────────────────────────────────────────────────────────────┐
  │  Пары Q → A  →  Люди оценивают  →  Reward Model  →  Модель       │
  │                    ответы             учится           учится      │
  │                                     оценивать        предпочитать │
  │                                     качество         лучшие ответы│
  └────────────────────────────────────────────────────────────────────┘


  ── Стадия 3: Оценка и итерации ────────────────────────────────────────

  Модель тестируют на бенчмарках (стандартизированных тестах) и в
  реальных диалогах, находят слабые места, корректируют данные и
  гиперпараметры, повторяют весь цикл.

  ┌────────────────────────────────────────────────────────────────────┐
  │  Бенчмарки ──→ Реальные тесты ──→ Корректировка ──→ ↻ Повтор    │
  └────────────────────────────────────────────────────────────────────┘


  ┌─ КЛЮЧЕВОЙ ИНСАЙТ ─────────────────────────────────────────────────┐
  │                                                                    │
  │  После обучения модель НЕ ХРАНИТ ТЕКСТ как базу данных.           │
  │  Она хранит миллиарды чисел, которые в совокупности определяют,   │
  │  как входной сигнал преобразуется в выходной.                      │
  │  Это ближе к мышечной памяти пианиста, чем к записной книжке.     │
  │                                                                    │
  └────────────────────────────────────────────────────────────────────┘

  Возьмём конкретный пример — ты пишешь:
  «Расскажи анекдот про кота»


╔══════════════════════════════════════════════════════════════════════════════╗
║  ШАГ 1. ВВОД ПОЛЬЗОВАТЕЛЯ                                                  ║
║                                                                              ║
║  "Расскажи анекдот про кота"                                                ║
║                                                                              ║
║  Это просто строка символов в Unicode. Модель пока ничего                    ║
║  с ней не может сделать — ей нужны числа.                                    ║
╚════════════════════════════╤═════════════════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════════════════╗
║  ШАГ 2. ТОКЕНИЗАЦИЯ                                                        ║
║                                                                              ║
║  Модель не работает с буквами и не работает с целыми словами.               ║
║  Она работает с ТОКЕНАМИ — фрагментами текста, которые лежат               ║
║  где-то между слогами и словами.                                             ║
║                                                                              ║
║  ╔═══════════════════════════════════════════════════════════════════════╗    ║
║  ║  АНАЛОГИЯ «АЗБУКА МОРЗЕ»                                           ║    ║
║  ║                                                                     ║    ║
║  ║  Представь, что ты шифруешь текст азбукой Морзе, но вместо точек   ║    ║
║  ║  и тире у тебя словарь из ~100 000 кусочков. Частые слова          ║    ║
║  ║  («the», «is», «что») — это один токен. Редкие слова разбиваются   ║    ║
║  ║  на части: «трансформер» → «транс» + «форм» + «ер».               ║    ║
║  ╚═══════════════════════════════════════════════════════════════════════╝    ║
║                                                                              ║
║  Как работает BPE (Byte-Pair Encoding):                                      ║
║  ──────────────────────────────────────                                      ║
║  Словарь модели — это таблица поиска, где каждому фрагменту текста           ║
║  сопоставлено число. Вместо хранения целых слов, BPE разбивает               ║
║  слова на частые подслова (фрагменты). Алгоритм выбирает                     ║
║  разбиение, которое минимизирует общее количество токенов. 
   Для слов, которых вообще нет в словаре, резервируется           
║  специальный токен [UNK] (unknown).                    
║                                                                              ║
║  Это позволяет обрабатывать даже незнакомые слова — они просто               ║
║  разобьются на известные части.                                              ║
║                                                                              ║
║  Пример:                                                                     ║
║  "Расскажи анекдот про кота"                                                ║
║      ↓ BPE tokenizer                                                        ║
   ["Рас", "скажи", " анек", "дот", " про", " кота"]                         ║
║      ↓ lookup в словаре                                                     ║
    Каждый токен получает числовой ID из словаря:
║   [4821,  9903,    7712,   3304,  1082,   8845]                              ║
║                                                                              ║
║  Специальные токены:                                                         ║
║  ──────────────────────────────────────                                      ║
║  Помимо обычных токенов, добавляются служебные:                              ║
║  [BOS] — начало последовательности (Begin of Sequence)                       ║
║  [EOS] — конец последовательности (End of Sequence)                          ║
║  [PAD] — заполнитель для выравнивания длины                                  ║
║  [UNK] — для слов, которых вообще нет в словаре (unknown)                   ║
║  Они дают модели дополнительный контекст о структуре ввода.                  ║
╚════════════════════════════╤═════════════════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════════════════╗
║  ШАГ 3. ЭМБЕДДИНГИ (Embeddings) — СЛОВА СТАНОВЯТСЯ ВЕКТОРАМИ               ║
║                                                                              ║
║  Каждый token ID превращается в ВЕКТОР — длинный массив чисел               ║
║  (обычно 4096–12288 значений). Это и есть эмбеддинг.                        ║
║                                                                              ║
║  ID 4821 ("Рас")    → [0.12, -0.45, 0.78, 0.03, ..., -0.21]  (4096 чисел) ║
║  ID 9903 ("скажи")  → [0.56,  0.11, -0.33, 0.89, ...,  0.44]              ║
║  ID 7712 (" анек")  → [-0.50, 1.70, 0.22, -0.18, ...,  0.09]              ║
║  ...и так для каждого токена                                                 ║
║                                                                              ║
║                                                                              ║
║  ЗАЧЕМ НУЖНЫ ЭМБЕДДИНГИ?                                                    ║
║  ────────────────────────                                                    ║
║  Числовой ID (например 4821) — это просто индекс, как номер                 ║
║  в гардеробе. Он ничего не говорит о смысле: ID 5000 ничем                  ║
║  не «ближе» к ID 5001 по значению.                                          ║
║                                                                              ║
║  Эмбеддинг же передаёт СМЫСЛ: похожие по значению слова получают            ║
║  похожие векторы (близкие точки в пространстве).                             ║
║                                                                              ║
║  ╔═══════════════════════════════════════════════════════════════════════╗    ║
║  ║  АНАЛОГИЯ «СМЫСЛОВАЯ БИБЛИОТЕКА»                                    ║    ║
║  ║                                                                     ║    ║
║  ║  Представь библиотеку, где книги расставлены не по алфавиту,        ║    ║
║  ║  а по содержанию. Книга про астрономию стоит рядом с книгой         ║    ║
║  ║  про физику звёзд, хотя их авторы начинаются на разные буквы.       ║    ║
║  ║  Эмбеддинг — это «адрес» слова в такой смысловой библиотеке,       ║    ║
║  ║  но в пространстве с тысячами измерений.                            ║    ║
║  ╚═══════════════════════════════════════════════════════════════════════╝    ║
║                                                                              ║
║                                                                              ║
║  ПОЧЕМУ ВЕКТОР?                                                              ║
║  ──────────────                                                              ║
║  Потому что у каждого слова 4096+ координат. Каждая координата —             ║
║  одно «измерение смысла». Какие-то координаты могут кодировать               ║
║  «одушевлённость», другие — «эмоциональную окраску», третьи —               ║
║  «абстрактность». Ни одна координата не подписана — они                      ║
║  возникают в процессе обучения.                                              ║
║                                                                              ║
║                                                                              ║
║  ПОЧЕМУ МАТРИЦЫ?                                                             ║
║  ───────────────                                                             ║
║  Потому что у нас не один вектор, а целая последовательность —               ║
║  столько, сколько токенов в запросе. Если запрос — 6 токенов,                ║
║  а каждый эмбеддинг — вектор из 4096 чисел, то вся входная                  ║
║  последовательность — это МАТРИЦА 6×4096.                                    ║
║  Матрица — это просто таблица, где каждая строка — один                      ║
║  токен-вектор.                                                               ║
║                                                                              ║
║  Таблица эмбеддингов:                                                        ║
║  ┌────────────┬─────────────────────────────────────────────────┐            ║
║  │ Token ID   │ Вектор (4096 чисел)                             │            ║
║  ├────────────┼─────────────────────────────────────────────────┤            ║
║  │ 4821       │ [ 0.12, −0.45,  0.78,  0.03, ..., −0.21]      │            ║
║  │ 9903       │ [ 0.56,  0.11, −0.33,  0.89, ...,  0.44]      │            ║
║  │ 7712       │ [−0.50,  1.70,  0.22, −0.18, ...,  0.09]      │            ║
║  │ 3304       │ [ 0.34, −0.88,  0.15,  0.62, ..., −0.55]      │            ║
║  │ 1082       │ [−0.17,  0.29, −0.71,  0.43, ...,  0.12]      │            ║
║  │ 8845       │ [ 0.91, −0.06,  0.58, −0.34, ...,  0.77]      │            ║
║  └────────────┴─────────────────────────────────────────────────┘            ║
║                                                                              ║
║  Изначально эмбеддинги — случайные числа. Осмысленные                        ║
║  представления формируются в процессе обучения модели.                        ║
║                                                                              ║
║  ВАЖНО: на этом этапе эмбеддинг одного и того же слова ОДИНАКОВ              ║
║  в любом контексте. Например "light" (свет) и "light" (лёгкий)              ║
║  пока имеют один и тот же вектор.                                            ║
║  Решение этой проблемы — задача трансформера (шаг 5).                        ║
╚════════════════════════════╤═════════════════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════════════════╗
║  ШАГ 4. ПОЗИЦИОННОЕ КОДИРОВАНИЕ (Positional Encoding)                       ║
║                                                                              ║
║  Эмбеддинги не знают ПОРЯДКА слов. Для модели «кот съел рыбу» и            ║
║  «рыба съела кота» пока неразличимы — оба набора токенов дали бы            ║
║  одинаковые эмбеддинги, только в другом порядке.                             ║
║                                                                              ║
║  Поэтому к каждому эмбеддингу ПРИБАВЛЯЕТСЯ вектор позиции —                 ║
║  уникальная «подпись», кодирующая место токена в последовательности.         ║
║                                                                              ║
║  Это отдельный набор эмбеддингов, представляющих положение                   ║
║  каждого токена. Он комбинируется (складывается) с эмбеддингом              ║
║  токена:                                                                     ║
║                                                                              ║
║  Эмбеддинг "скажи":     [0.56,  0.11, -0.33, ...]     (что это за слово)   ║
║  Позиционный вектор:   + [0.02, -0.15,  0.33, ...]     (где оно стоит)     ║
║                        ─────────────────────────────                         ║
║  Итого:                = [0.58, -0.04,  0.00, ...]     (слово + позиция)    ║
║                                                                              ║
║  Без этого модель не отличила бы "кот ел рыбу" от "рыба ела кота" —        ║
║  набор токенов одинаков, отличается только порядок.                          ║
║                                                                              ║
║                                                                              ║
║  РЕЗУЛЬТАТ ШАГОВ 2–4:                                                        ║
║  ─────────────────────                                                       ║
║  Матрица 6 × 4096 — готовый вход для трансформера.                          ║
║  Каждая строка — один токен, превращённый в вектор,                         ║
║  обогащённый информацией о позиции.                                          ║
║                                                                              ║
║  ┌───────┬──────────────────────────────────────────────────────┐            ║
║  │ t₁    │ [0.14, −0.30,  1.11, ..., −0.09]  ← "Рас" + pos₁  │            ║
║  │ t₂    │ [0.58, −0.04,  0.00, ...,  0.56]  ← "скажи" + pos₂│            ║
║  │ t₃    │ [−0.48, 1.55,  0.55, ...,  0.18]  ← " анек" + pos₃│            ║
║  │ t₄    │ [0.36, −0.73,  0.48, ..., −0.43]  ← "дот" + pos₄  │            ║
║  │ t₅    │ [−0.15, 0.14, −0.38, ...,  0.24]  ← " про" + pos₅ │            ║
║  │ t₆    │ [0.93,  0.09,  0.91, ...,  0.89]  ← " кота" + pos₆│            ║
║  └───────┴──────────────────────────────────────────────────────┘            ║
║              ↑                                    ↑                          ║
║          6 строк                           4096 столбцов                     ║
╚════════════════════════════╤═════════════════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════════════════╗
║  ШАГ 5. ТРАНСФОРМЕР (Transformer) — СЛОЙ ЗА СЛОЕМ                           ║
║                                                                              ║
║  Ядро модели. Стопка одинаковых слоёв (в современных моделях                ║
║  80–120 слоёв). Каждый слой содержит:                                        ║
║                                                                              ║
║    1) Self-Attention  — токены «разговаривают» друг с другом                ║
║    2) Feed-Forward    — каждый токен «думает» в одиночку                    ║
║    3) Layer Norm + Residual Connections — стабилизация                       ║
║                                                                              ║
║  Главное преимущество трансформеров — способность обрабатывать               ║
║  длинные зависимости в тексте (связь между словами, которые                  ║
║  далеко друг от друга) и делать это параллельно, а не                        ║
║  последовательно, как старые архитектуры (RNN/LSTM).                         ║
║                                                                              ║
║                                                                              ║
║  ┌────────────────────────────────────────────────────────────────────┐      ║
║  │                                                                    │      ║
║  │  ┌──────────────────────────────────────────────────────────────┐  │      ║

║  │  │    ▼                              │                          │  │      ║

║  ═══════════════════════════════════════════════════════════════════════      ║
║  5A. SELF-ATTENTION (механизм внимания) — ЯДРО ТРАНСФОРМЕРА                  ║
║  ═══════════════════════════════════════════════════════════════════════      ║
║                                                                              ║
║  Это самая важная часть. Каждый токен «смотрит» на все остальные             ║
║  токены и определяет, какие из них важны для текущего контекста.             ║
║                                                                              ║
║  "кота" обращает внимание на "анекдот" и "расскажи"                         ║
║  → понимает: нужен юмор про кота                                            ║
║                                                                              ║
║  ╔═══════════════════════════════════════════════════════════════════════╗    ║
║  ║  АНАЛОГИЯ «ВЕЧЕРИНКА»                                               ║    ║
║  ║                                                                     ║    ║
║  ║  Представь вечеринку, где каждый гость — это токен.                 ║    ║
║  ║  Механизм внимания — это то, как каждый гость решает,              ║    ║
║  ║  к кому прислушиваться.                                             ║    ║
║  ║                                                                     ║    ║
║  ║  Каждый гость создаёт три визитки:                                  ║    ║
║  ║  • Query (Q)  — «Что мне интересно?»                               ║    ║
║  ║  • Key (K)    — «Чем я могу быть полезен?»                         ║    ║
║  ║  • Value (V)  — «Вот мой вклад»                                    ║    ║
║  ║                                                                     ║    ║
║  ║  Каждый гость СРАВНИВАЕТ свой Query с Key каждого другого гостя.   ║    ║
║  ║  Чем лучше совпадение — тем внимательнее он слушает этого гостя.   ║    ║
║  ║  Потом он ВЗВЕШЕННО СУММИРУЕТ Value всех остальных —               ║    ║
║  ║  пропорционально тому, насколько внимательно он их слушал.          ║    ║
║  ╚═══════════════════════════════════════════════════════════════════════╝    ║
║                                                                              ║
║  Как это работает технически:                                                ║
║  ─────────────────────────────                                               ║
║                                                                              ║
║  Матрица эмбеддингов (X) пропускается через три независимых                  ║
║  линейных слоя (матричных умножения), получая:                               ║
║                                                                              ║
║    Q = X · W_q     (Query  — "что я ищу?")                                  ║
║    K = X · W_k     (Key    — "что я содержу?")                              ║
║    V = X · W_v     (Value  — "что я отдам?")                                ║
║                                                                              ║
║    W_q, W_k, W_v — это матрицы весов (ПАРАМЕТРЫ!), которые                  ║
║    модель выучила при обучении.                                              ║
║                                                                              ║
║  Далее четыре шага:                                                          ║
║                                                                              ║
║  1) Вычисляем скалярное произведение Q × K^T                                ║
║     → матрица «оценок внимания» (attention scores)                          ║
║     Чем больше произведение — тем больше сходство                            ║
║     между двумя токенами                                                     ║
║                                                                              ║
║  2) Масштабирование: делим на √d (размерность ключа)                        ║
║     Без этого большие числа после умножения дадут слишком                    ║
║     «острое» распределение, что приведёт к проблеме                         ║
║     исчезающего градиента при обучении                                       ║
║                                                                              ║
║  3) Softmax — превращаем оценки в веса (0..1),                              ║
║     которые в сумме дают 1                                                   ║
║                                                                              ║
║  4) Умножаем веса на V → получаем новые                                      ║
║     контекстуализированные эмбеддинги                                        ║
║     (теперь каждый токен «впитал» информацию                                ║
║      от релевантных соседей)                                                 ║
║                                                                              ║
║                                                                              ║
║  Итого формула:                                                              ║
║  ┌──────────────────────────────────────────────────────────────────┐        ║
║  │                                                                  │        ║
║  │  Attention(Q, K, V) = softmax(Q · K^T / √d_k) · V              │        ║
║  │                                                                  │        ║
║  └──────────────────────────────────────────────────────────────────┘        ║
║                                                                              ║
║                                                                              ║
║  Пример матрицы весов внимания для нашего запроса:                           ║
║                                                                              ║
║              Рас   скажи  анек   дот    про   кота                          ║
║  Рас     [  0.25   0.15   0.10   0.10   0.10  0.30 ]                       ║
║  скажи   [  0.10   0.20   0.25   0.15   0.10  0.20 ]                       ║
║  анек    [  0.05   0.30   0.20   0.35   0.05  0.05 ]                       ║
║  дот     [  0.05   0.10   0.40   0.25   0.05  0.15 ]                       ║
║  про     [  0.05   0.15   0.10   0.10   0.15  0.45 ]                       ║
║  кота    [  0.10   0.30   0.25   0.10   0.05  0.20 ]                       ║
║                                                                              ║
║  Читаем: «про» сильнее всего «прислушивается» к «кота» (0.45),             ║
║  потому что контекст «про кота» создаёт сильную связь.                      ║
║  «кота» прислушивается к «скажи» (0.30) и «анек» (0.25) —                  ║
║  модель понимает, что нужен юмор.                                            ║
║                                                                              ║
║                                                                              ║
║  Self-attention означает, что Q, K, V получены из ОДНОЙ и той же            ║
║  входной матрицы — модель ищет закономерности внутри своих же                ║
║  входных данных.                                                             ║
║                                                                              ║
║                                                                              ║
║  МАСКИРОВАННОЕ ВНИМАНИЕ (в декодерах):                                       ║
║  ──────────────────────────────────────                                      ║
║  В декодерах (GPT, Claude) используется маскированный self-attention:        ║
║  каждый токен видит только СЕБЯ и ПРЕДЫДУЩИЕ токены. Будущие позиции        ║
║  маскируются через нижнетреугольную матрицу (замена на −∞ →                  ║
║  после softmax даёт 0). Модель не может «подглядывать вперёд».              ║
║                                                                              ║
║  Пример маски (✓ = видит, ✗ = замаскировано):                               ║
║                                                                              ║
║              Рас   скажи  анек   дот    про   кота                          ║
║  Рас     [   ✓      ✗      ✗      ✗      ✗     ✗  ]                        ║
║  скажи   [   ✓      ✓      ✗      ✗      ✗     ✗  ]                        ║
║  анек    [   ✓      ✓      ✓      ✗      ✗     ✗  ]                        ║
║  дот     [   ✓      ✓      ✓      ✓      ✗     ✗  ]                        ║
║  про     [   ✓      ✓      ✓      ✓      ✓     ✗  ]                        ║
║  кота    [   ✓      ✓      ✓      ✓      ✓     ✓  ]                        ║
║                                                                              ║
║                                                                              ║
║  ═══════════════════════════════════════════════════════════════════════      ║
║  5B. MULTI-HEAD ATTENTION (многоголовое внимание)                            ║
║  ═══════════════════════════════════════════════════════════════════════      ║
║                                                                              ║
║  На вечеринке каждый гость создаёт не одну тройку визиток (Q, K, V),        ║
║  а 32, 64 или даже 128 комплектов — по числу «голов внимания».              ║
║                                                                              ║
║  Каждая голова специализируется на своём типе связей:                        ║
║  • Одна голова следит за СИНТАКСИСОМ (подлежащее → сказуемое)               ║
║  • Другая — за СЕМАНТИЧЕСКОЙ БЛИЗОСТЬЮ (анекдот ↔ кота)                    ║
║  • Третья — за ДАЛЬНИМИ ССЫЛКАМИ (местоимение → существительное)            ║
║  • Четвёртая — за ТОНОМ (расскажи → юмор)                                   ║
║                                                                              ║
║  Размерность каждой головы = размер эмбеддинга / кол-во голов               ║
║  (чтобы суммарный размер не менялся).                                        ║
║  Пример: 4096 / 64 головы = 64 чисел на голову                              ║
║                                                                              ║
║  Схема:                                                                      ║
║  ┌────────────────────────────────────────────────────────────────┐          ║
║  │  Вход (6×4096)                                                 │          ║
║  │    │                                                           │          ║
║  │    ├──→ [Голова 1: Q₁·K₁ᵀ·V₁] ──┐                            │          ║
║  │    ├──→ [Голова 2: Q₂·K₂ᵀ·V₂] ──┤                            │          ║
║  │    ├──→ [Голова 3: Q₃·K₃ᵀ·V₃] ──┼──→ Конкатенация ──→ W_o   │          ║
║  │    ├──→  ...                      │                            │          ║
║  │    └──→ [Голова 64: ...]     ─────┘                            │          ║
║  │                                           │                    │          ║
║  │                                           ▼                    │          ║
║  │                                    Выход (6×4096)              │          ║
║  └────────────────────────────────────────────────────────────────┘          ║
║                                                                              ║
║  Результаты всех голов конкатенируются и пропускаются через                  ║
║  линейный слой (W_o), который комбинирует информацию от каждой               ║
║  головы в единое представление.                                              ║
║                                                                              ║
║                                                                              ║
║  ═══════════════════════════════════════════════════════════════════════      ║
║  5C. FEED-FORWARD NETWORK (FFN) — «ТИХОЕ РАЗМЫШЛЕНИЕ»                       ║
║  ═══════════════════════════════════════════════════════════════════════      ║
║                                                                              ║
║  После внимания каждый токен (уже обогащённый информацией от других)         ║
║  проходит через простую двухслойную нейросеть.                               ║
║                                                                              ║
║  Она одинакова для всех токенов, но применяется к каждому НЕЗАВИСИМО.        ║
║                                                                              ║
║  Если self-attention — это «разговор с другими», то FFN — это                ║
║  «внутренний монолог», где токен обрабатывает собранную информацию.          ║
║                                                                              ║
║  FFN(x) = GeLU(x · W₁ + b₁) · W₂ + b₂                                     ║
║                                                                              ║
║  Размерности:  4096  →  16384  →  4096                                      ║
║                (вход)  (расширение в 4×)  (сжатие обратно)                   ║
║                                                                              ║
║  FFN часто описывают как ХРАНИЛИЩЕ ЗНАНИЙ модели: именно здесь,              ║
║  в весах этих слоёв, закодированы факты вроде «Париж — столица              ║
║  Франции» или «анекдот — это форма юмора».                                  ║
║                                                                              ║
║  Задача FFN — добавить дополнительные изучаемые параметры, чтобы            ║
║  контекстуальные эмбеддинги были хорошо разделены и распределены             ║
║  в пространстве.                                                             ║
║                                                                              ║
║                                                                              ║
║  ═══════════════════════════════════════════════════════════════════════      ║
║  5D. LAYER NORMALIZATION + SKIP CONNECTIONS                                  ║
║  ═══════════════════════════════════════════════════════════════════════      ║
║                                                                              ║
║  Layer Norm — нормализует значения в каждом слое, не позволяя им            ║
║  стать слишком большими или маленькими. Без этого глубокие модели            ║
║  страдают от исчезающего или взрывающегося градиента (обучение               ║
║  ломается).                                                                  ║
║                                                                              ║
║  Skip Connections (остаточные связи) — вход слоя ПРИБАВЛЯЕТСЯ               ║
║  к его выходу (как в ResNet):                                                ║
║                                                                              ║
║    output = LayerNorm(x + Attention(x))                                      ║
║    output = LayerNorm(output + FFN(output))                                  ║
║                                                                              ║
║  ╔═══════════════════════════════════════════════════════════════════════╗    ║
║  ║  АНАЛОГИЯ «ЛИФТ»                                                    ║    ║
║  ║                                                                     ║    ║
║  ║  Это как лифт, который идёт с первого по 96-й этаж без потери       ║    ║
║  ║  пассажиров — каждый этаж может что-то добавить, но ничего          ║    ║
║  ║  не забирает. Если слой ничего полезного не добавил,                ║    ║
║  ║  информация всё равно проходит.                                     ║    ║
║  ╚═══════════════════════════════════════════════════════════════════════╝    ║
║                                                                              ║
║  Это позволяет градиенту «протекать» через слои напрямую, не затухая —      ║
║  критично для обучения моделей с десятками/сотнями слоёв.                    ║
║                                                                              ║
║  Современный подход (pre-layer norm):                                        ║
║  нормализация ставится ДО attention и FFN                                    ║
║  (а не после, как в оригинальной статье "Attention Is All You Need").        ║
║                                                                              ║
║                                                                              ║
║  ═══════════════════════════════════════════════════════════════════════      ║
║  5E. ЧТО ДЕЛАЮТ РАЗНЫЕ СЛОИ                                                 ║
║  ═══════════════════════════════════════════════════════════════════════      ║
║                                                                              ║
║  Это повторяется N раз (слой за слоем), каждый раз углубляя                  ║
║  понимание контекста:                                                        ║
║                                                                              ║
║  • Ранние слои (1–20):  улавливают простые паттерны — грамматика,           ║
║                          согласование, базовый синтаксис                      ║
║  • Средние слои (20–60): семантика — смысловые связи, тематика,             ║
║                          отношения между понятиями                           ║
║  • Глубокие слои (60+):  абстрактное мышление — логика, рассуждения,        ║
║                          планирование ответа, стиль                          ║
║                                                                              ║
║  После прохождения всех слоёв эмбеддинги становятся                          ║
║  КОНТЕКСТУАЛИЗИРОВАННЫМИ — теперь "light" в контексте                        ║
║  "light bag" и "light in the room" имеют РАЗНЫЕ векторы.                    ║
╚════════════════════════════╤═════════════════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════════════════╗
║  ШАГ 6. ЛОГИТЫ (Logits) — ПРОЕКЦИЯ В СЛОВАРЬ                               ║
║                                                                              ║
║  На выходе трансформера — обновлённая матрица. Для генерации                 ║
║  следующего слова модель берёт вектор ПОСЛЕДНЕГО токена                      ║
║  (именно он аккумулировал весь контекст через attention)                     ║
║  и умножает его на матрицу весов размером [d × vocab_size].                  ║
║                                                                              ║
║  По сути, это та же матрица эмбеддингов, но в обратную сторону:              ║
║  из вектора 4096 → вектор 100 000 (по числу токенов в словаре).             ║
║                                                                              ║
║  Результат — вектор длиной = размер словаря:                                 ║
║                                                                              ║
║  "Идёт"    →  5.8        ← высокий логит = вероятный кандидат               ║
║  "Жил"     →  5.2                                                            ║
║  "Кот"     →  4.1                                                            ║
║  "Однажды" →  3.9                                                            ║
║  "The"     → -2.1        ← низкий логит = маловероятно                      ║
║  "xyz"     → -9.4        ← почти невозможно                                ║
║  ... (100 000+ оценок — по одной на каждый токен в словаре)                 ║
║                                                                              ║
║  Это ещё НЕ вероятности, просто «сырые» числа (логиты).                     ║
╚════════════════════════════╤═════════════════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════════════════╗
║  ШАГ 7. СЭМПЛИРОВАНИЕ (выбор следующего токена)                             ║
║                                                                              ║
║  После получения логитов нужно выбрать один токен из                         ║
║  100 000+ кандидатов. Объём данных огромен, поэтому                          ║
║  применяется цепочка фильтров для сужения выбора:                            ║
║                                                                              ║
║                                                                              ║
║  7.1  Temperature (температура)                                              ║
║  ──────────────────────────────                                              ║
║  Каждый логит делится на temperature:                                         ║
║    новый_логит = логит / temperature                                         ║
║                                                                              ║
║  Низкая (0.1) → разница усиливается → предсказуемый ответ                   ║
║  Нормальная (1.0) → без изменений                                            ║
║  Высокая (2.0) → разница сглаживается → креативный ответ                    ║
║                                                                              ║
║  При T→0 модель почти всегда выбирает самый вероятный токен.                ║
║  При T→∞ выбор становится почти случайным.                                   ║
║                                                                              ║
║                         │                                                    ║
║                         ▼                                                    ║
║  7.2  Top-K                                                                  ║
║  ──────────                                                                  ║
║  Оставляем только K самых вероятных токенов.                                 ║
║  (topK=64 → берём 64, остальные ~99 936 отбрасываем)                        ║
║                                                                              ║
║  Фильтрация по количеству: оставляем фиксированное число                    ║
║  лучших кандидатов на основе их логитов.                                      ║
║                                                                              ║
║                         │                                                    ║
║                         ▼                                                    ║
║  7.3  Top-P (Nucleus Sampling)                                               ║
║  ──────────────────────────────                                              ║
║  Из оставшихся берём минимальный набор, чья суммарная                        ║
║  вероятность ≥ порога (topP=0.95 → 95%).                                     ║
║                                                                              ║
║  Количество кандидатов плавающее — в одном случае останется                  ║
║  5 токенов, в другом 50, в зависимости от того, насколько                    ║
║  модель «уверена» в следующем слове.                                         ║
║                                                                              ║
║                         │                                                    ║
║                         ▼                                                    ║
║  7.4  Softmax → Вероятности                                                  ║
║  ──────────────────────────                                                  ║
║  Преобразуем отфильтрованные логиты в вероятности                            ║
║  (сумма = 100%):                                                             ║
║                                                                              ║
║  "Идёт"    → 42%                                                             ║
║  "Жил"     → 31%                                                             ║
║  "Кот"     → 18%                                                             ║
║  "Однажды" →  9%                                                             ║
║                                                                              ║
║                         │                                                    ║
║                         ▼                                                    ║
║  7.5  Случайный выбор (sampling)                                             ║
║  ────────────────────────────────                                            ║
║  Бросаем «кубик» с учётом вероятностей                                      ║
║  (выборка из полиномиального распределения).                                 ║
║                                                                              ║
║  Результат: "Идёт"                                                           ║
╚════════════════════════════╤═════════════════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════════════════╗
║  ШАГ 8. ПРОВЕРКА ОСТАНОВКИ                                                  ║
║                                                                              ║
║  Сгенерированный токен = [EOS] (конец)?    → СТОП                           ║
║  Достигнут maxOutputTokens?                 → СТОП                           ║
║  Встретился stop sequence?                  → СТОП                           ║
║  Ничего из этого?                           → ПРОДОЛЖАЕМ ↓                  ║
╚════════════════════════════╤═════════════════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════════════════╗
║  ШАГ 9. АВТОРЕГРЕССИОННЫЙ ЦИКЛ                                              ║
║                                                                              ║
║  Выбранный токен добавляется к входной последовательности:                   ║
║                                                                              ║
║  Итерация 1: "Расскажи анекдот про кота"          → "Идёт"                 ║
║  Итерация 2: "Расскажи анекдот про кота Идёт"     → "кот"                  ║
║  Итерация 3: "Расскажи анекдот про кота Идёт кот" → "по"                   ║
║  ...                                                                         ║
║                                                                              ║
║  Каждая итерация проходит ЗАНОВО шаги 3 → 4 → 5 → 6 → 7 → 8               ║
║  (с учётом KV-cache для ускорения)                                           ║
║                                                                              ║
║                                                                              ║
║  KV-Cache (оптимизация скорости):                                            ║
║  ─────────────────────────────────                                           ║
║  При генерации каждого нового токена модель пересчитывает                    ║
║  attention для всей последовательности. Но K и V для уже                     ║
║  обработанных токенов НЕ МЕНЯЮТСЯ — их можно кэшировать.                    ║
║                                                                              ║
║  Новый токен вычисляет свой Q и сравнивает его с сохранёнными               ║
║  K и V, вместо того чтобы пересчитывать всё с нуля.                         ║
║  Это значительно ускоряет генерацию.                                         ║
║                                                                              ║
║  Без KV-cache: O(n²) вычислений на каждый новый токен                       ║
║  С KV-cache:   O(n) вычислений на каждый новый токен                        ║
╚════════════════════════════╤═════════════════════════════════════════════════╝
                             │
                             ▼
╔══════════════════════════════════════════════════════════════════════════════╗
║  ШАГ 10. ДЕТОКЕНИЗАЦИЯ                                                      ║
║                                                                              ║
║  Набранные токены склеиваются обратно в текст:                               ║
║                                                                              ║
║  ["Идёт", " кот", " по", " крыше", ",", " видит", "..."]                   ║
║       ↓                                                                      ║
║  "Идёт кот по крыше, видит..."                                               ║
║                                                                              ║
║  Этот текст отдаётся пользователю.                                           ║
║  При streaming — токен за токеном по мере генерации                          ║
║  (именно поэтому ответ «печатается» постепенно).                            ║

  Вся работа трансформера — это последовательные умножения матриц.

  Почему именно матрицы? Потому что МАТРИЧНОЕ УМНОЖЕНИЕ — это
  универсальный язык преобразований. Когда ты умножаешь вектор слова
  на матрицу весов, ты вращаешь и растягиваешь его в пространстве
  смыслов.

  Каждый слой трансформера — это серия таких вращений и растяжений,
  которые постепенно превращают «сырые» координаты слова в координаты,
  кодирующие ответ.

  ╔═══════════════════════════════════════════════════════════════════════╗
  ║  АНАЛОГИЯ «ПЕРЕРИСОВКА КАРТЫ»                                        ║
  ║                                                                     ║
  ║  Представь, что каждый токен — это точка на карте.                   ║
  ║  Матричное умножение — это как перерисовка карты:                    ║
  ║  страны меняют форму, некоторые сближаются, некоторые отдаляются.    ║
  ║                                                                     ║
  ║  Через 96 таких перерисовок точка, которая начиналась как           ║
  ║  «Расскажи», оказывается в той области пространства,                ║
  ║  которая кодирует «Идёт кот по крыше...»                           ║
  ╚═══════════════════════════════════════════════════════════════════════╝

  Все основные операции трансформера — это матричные умножения:

  ┌────────────────────────────────────────────────────────────────────┐
  │  Операция                    │  Что происходит                    │
  ├────────────────────────────────────────────────────────────────────┤
  │  Эмбеддинг (шаг 3)          │  ID × Таблица = вектор смысла      │
  │  Создание Q, K, V (шаг 5A)  │  X × W_q, X × W_k, X × W_v       │
  │  Оценки внимания (шаг 5A)   │  Q × K^T = матрица сходства        │
  │  Взвешенная сумма (шаг 5A)  │  Веса × V = новые эмбеддинги       │
  │  Объединение голов (шаг 5B)  │  [head₁;...;headₙ] × W_o         │
  │  FFN (шаг 5C)               │  x × W₁, затем × W₂               │
  │  Логиты (шаг 6)             │  вектор × Таблица^T = оценки       │
  └────────────────────────────────────────────────────────────────────┘

  GPU/TPU оптимизированы именно для параллельных матричных умножений —
  поэтому трансформеры так хорошо масштабируются на специализированном
  оборудовании.

  Трансформеры бывают трёх типов в зависимости от того,
  какие блоки используются:


  ┌─────────────────────────────────────────────────────────────────────┐
  │  A. Энкодер (Encoder)                                               │
  │                                                                     │
  │  Видит ВСЮ входную последовательность целиком (двунаправленный).    │
  │  Задача — создать контекстуализированные эмбеддинги                 │
  │  для последующих задач: классификация текста,                       │
  │  распознавание сущностей, поиск и т.д.                              │
  │                                                                     │
  │  Пример: BERT, RoBERTa                                              │
  │  Применение: "Этот отзыв положительный или отрицательный?"          │
  │              Поиск по документам, NER, вопрос-ответ                  │
  └─────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────┐
  │  B. Декодер (Decoder)                                               │
  │                                                                     │
  │  Видит только текущий и ПРЕДЫДУЩИЕ токены (однонаправленный,       │
  │  будущие замаскированы). Генерирует текст токен за токеном.         │
  │  Именно эта архитектура используется в ChatGPT, Claude, Gemini.    │
  │                                                                     │
  │  Пример: GPT, Claude, LLaMA, Gemini                                 │
  │  Применение: генерация текста, диалоги, написание кода,             │
  │              рассуждения, решение задач                              │
  └─────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────┐
  │  C. Энкодер-Декодер                                                 │
  │                                                                     │
  │  Энкодер читает вход целиком → создаёт промежуточное                │
  │  представление → декодер генерирует выход на его основе.             │
  │                                                                     │
  │  Декодер при этом использует cross-attention:                       │
  │  K и V берутся из ЭНКОДЕРА, Q — из ДЕКОДЕРА.                       │
  │  (сравни с self-attention, где Q, K, V — из одного входа)          │
  │                                                                     │
  │  Пример: T5, BART, оригинальный Transformer                         │
  │  Применение: перевод, суммаризация, ответы на вопросы               │
  └─────────────────────────────────────────────────────────────────────┘

  Параметры (weights) — числа внутри матриц нейросети; настраиваются
      при обучении; определяют поведение модели.

  Токен — минимальная единица текста для модели; фрагмент между
      слогом и словом.

  BPE (Byte-Pair Encoding) — алгоритм токенизации; разбивает текст
      на частые подслова.

  Эмбеддинг (embedding) — вектор чисел, кодирующий смысл токена
      в многомерном пространстве.

  Вектор — упорядоченный набор чисел; координаты точки в пространстве.

  Матрица — таблица чисел; набор векторов, уложенных в строки.

  Позиционное кодирование — вектор, добавляемый к эмбеддингу для
      передачи информации о порядке слов.

  Self-Attention — механизм, где каждый токен определяет важность
      всех остальных токенов для себя.

  Query (Q) — «что я ищу?» — проекция токена для сравнения с другими.

  Key (K) — «что я содержу?» — проекция токена для сравнения
      с запросами других.

  Value (V) — «что я отдам?» — фактическая информация, которую
      токен передаёт другим.

  Multi-Head Attention — несколько параллельных блоков attention,
      каждый специализируется на своём типе связей.

  Feed-Forward Network (FFN) — двухслойная нейросеть, применяемая
      к каждому токену независимо; хранилище знаний.

  Layer Norm — нормализация значений внутри слоя для стабильности
      обучения.

  Residual Connection (Skip Connection) — обходной путь, прибавляющий
      вход слоя к его выходу; предотвращает потерю информации.

  Логиты (logits) — сырые оценки для каждого токена словаря
      перед преобразованием в вероятности.

  Softmax — функция, превращающая логиты в вероятности (сумма = 1).

  Temperature — параметр, управляющий «креативностью» генерации.

  Top-K — фильтр: оставить только K самых вероятных кандидатов.

  Top-P (nucleus sampling) — фильтр: оставить минимальный набор
      кандидатов, покрывающий P% вероятности.

  KV-Cache — кэширование Key и Value для ранее обработанных токенов;
      ускоряет генерацию.

  Авторегрессия — генерация по одному токену за раз, где каждый
      новый токен зависит от всех предыдущих.

  Loss — функция ошибки; разница между предсказанием модели
      и правильным ответом.

  Backpropagation — алгоритм обратного распространения ошибки;
      вычисляет, как изменить каждый параметр для уменьшения loss.

  RLHF — Reinforcement Learning from Human Feedback; метод
      дообучения модели на основе человеческих оценок.

  Reward Model — модель, обученная предсказывать человеческие
      предпочтения; направляет обучение основной модели.

  Cross-Attention — внимание, где Q из одного источника,
      а K и V из другого (используется в энкодер-декодерах).

  GeLU — функция активации в FFN; плавная версия ReLU.
